# Recovery-model validation — nonlinear constant-force solver

Context: the July 2026 change replaced the linear `time = firstRepTime ×
capacityFraction` rep-time predictor with a nonlinear constant-force
force-duration solver (`predictRepTimes` in `src/model/fatigue.js`), now
shared by the live forecast and the personal recovery-tau fitter. That
invalidated the old LOO-CV numbers (retired linear predictor) and put the
recovery **gap** (observed − predicted rep-2/rep-1 ratio) on a new scale.

Every number below is regenerated by
`src/model/__tests__/recoveryModel.validation.test.js` (self-skips in CI;
point `RECOVERY_VALIDATION_JSON` at an export of your multi-rep sets to
re-run). Dataset: ~5 months of the author's real sessions — Micro 90,
Crusher 72 mature test sets (a "mature" test set has ≥5 prior sets for the
grip; Prime excluded — too few sets). The harness reuses the exact
production code, including the deload detector's own `recentGapHeldOut`.

## 1. Personalization still helps (~10% out-of-sample)

Time-separated, forward-chained (expanding-window) holdout — the personal
taus for each test set are fit **only on sessions from strictly earlier
calendar days**, then scored on that set.

| grip    | mature sets | RMSE personal | RMSE population | Δ       |
|---------|-------------|---------------|-----------------|---------|
| Micro   | 90          | 13.99 s       | 15.52 s         | **−9.8%** |
| Crusher | 72          | 31.03 s       | 34.31 s         | **−9.6%** |

Personal recovery taus beat the population prior out-of-sample on both
grips. **Keep personalization** (with the existing conservative Bayesian
shrinkage for sparse grips).

Production now matches this discipline. `deload.js` previously fit taus on
the **entire** history and then scored the latest sessions with those same
taus — the sessions being evaluated leaked into their own baseline (worst
on sparse grips, where the recent window is a big fraction of the fit),
partly masking a real dip. `recentGapHeldOut` now fits each grip's baseline
only on sessions **before** the recent window and scores the recent ones
held-out.

## 2. Two different statistics → two separate constants

The app reads the gap through two different filters, which have different
noise. They are now **separate constants**, each calibrated to its own
statistic (they happen to coincide numerically on this data — not by
construction):

| statistic | used by | measured noise | constant |
|-----------|---------|----------------|----------|
| 3-session **smoothed** gap | recovery chart band + coaching early-warn | std 0.16 (Micro) / 0.25 (Crusher) | `GAP_NOISE_BAND = 0.15` (`recoveryDynamics.js`) |
| 2-session **mean** gap (held-out) | cross-grip deload gate | std 0.15 (Micro) / 0.26 (Crusher) | `DELOAD_GAP_TRIGGER = 0.15` (`deload.js`) |

Raw per-session gap std is 0.23 (Micro) / 0.38 (Crusher); ~56% / 74% of raw
dots fall outside ±0.10 and ~46% / 60% outside ±0.15 — which is why the
chart is trend-first (read the smoothed line, not the dots) and the old
±0.10 (tuned to the linear predictor) was ~half the real noise.

Both gap statistics are centered **positive** (2-session mean +0.09 Micro /
+0.14 Crusher): the nonlinear model slightly under-predicts this user's
recovery. So `DELOAD_GAP_TRIGGER = −0.15` is ~1–1.5σ **below the user's own
positive baseline** on the 2-session statistic — a beyond-noise systemic
dip, not scatter.

## 3. Deload gate is very conservative (fired 0% here)

The gate fires only when **every** measured grip's held-out 2-session mean
is below `−DELOAD_GAP_TRIGGER` (a cross-grip AND gate), with a lifting-volume
spike required for "strong". Simulated over 47 eligible ref dates:

| trigger | historical cross-grip fire rate |
|---------|--------------------------------|
| −0.10   | 0%                             |
| −0.15 (production) | 0%                    |
| −0.20   | 0%                             |

This athlete never hit a systemic deload condition in the window, so the
data can't distinguish trigger values on the "should-fire" side — the
trigger is set from the statistic's noise scale, not observed deloads. The
value is stable across −0.10…−0.20, so −0.15 changes nothing on real data
while being honestly derived. The synthetic `deload.test.js` fixtures (a
collapsed t2/t1 ≈ 0.33 → gap ≈ −0.6, far below any trigger) still fire.

## 4. Readiness gauge is a deliberate early runway

`deloadStatus` softens the light **before** the hard cross-grip deload so a
deload never appears out of nowhere. Its scale is now tied to the trigger
rather than a pre-widening magic number: `DELOAD_PRESSURE_SCALE =
DELOAD_GAP_TRIGGER` (full pressure at avgGap = −0.15, the deload line), and
`DELOAD_YELLOW_AT = 0.5` lights yellow at avgGap ≈ −0.075 — about halfway
down the runway. Because avgGap normally sits positive (see §2), the gauge
stays green unless cross-grip recovery genuinely drifts negative; yellow
inside the chart's ±0.15 band is intentional (a heads-up, not the
decision). Stale "±0.10" comments were corrected.
